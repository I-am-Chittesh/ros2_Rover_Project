# Rover project using ros2

## 1. SYSTEM CONTEXT & HARDWARE SPECIFICATIONS

### The Physical Robot (The Body)
- **Kinematics:** Differential drive configuration (2 independent active drive wheels + 1 passive caster wheel).
- **Primary Compute:** ESP32 microcontroller running embedded micro-ROS firmware communicating over a local Wi-Fi network.
- **Actuators:** DC motors controlled via an H-bridge motor driver (e.g., L298N/TB6612FNG) mapped to hardware PWM signals on the ESP32.
- **Sensors:** A localized discrete array of IR/Ultrasonic distance sensors positioned on the front and sides. 
- **CRITICAL HARDWARE CONSTRAINT:** The physical robot has NO hardware LiDAR and NO cameras. It relies entirely on discrete range sensors.

### The Simulation Host (The Brain & Map)
- **Environment:** ROS 2 Humble running inside Windows Subsystem for Linux (WSL) with native WSLg graphics bridging enabled.
- **Visualization:** RViz2 used for active data visualization (Transform trees, maps, sensor vectors).
- **Physics Engine:** Gazebo Classic used as the 3D physics simulator.

---

## 2. THE SIM-TO-REAL BRIDGE ARCHITECTURE

To circumvent the hardware lack of a physical LiDAR, the project splits the software stack into a "Digital Twin" closed-loop system:

1. **The Simulation Cheat-Code:** The robot's URDF model contains a virtual 2D LiDAR plugin (`gazebo_ros_ray_sensor`). This allows standard ROS 2 mapping and navigation nodes to run seamlessly inside the simulated environment.
2. **Autonomous Navigation Stack:** The host machine runs `slam_toolbox` to generate a 2D occupancy grid map, and the `nav2` (Navigation2) stack to calculate path planning (Global/Local costmaps) from origin to destination.
3. **Shadow Driving Execution:** The velocity commands (`/cmd_vel`) generated by Nav2 in the simulation are streamed over Wi-Fi to the physical ESP32 via micro-ROS. The physical wheels spin to mirror the simulation.
4. **The Reality Patch (Low-Level Override Loop):** Due to real-world wheel slippage and dynamic obstacles (unmapped objects like shoes/people), the physical rover cannot drive blindly. The ESP32 constantly polls its physical IR/Ultrasonic sensor array. If a distance threshold is breached (indicating an imminent collision or severe drift), the ESP32 instantly intercepts the incoming `/cmd_vel` command, executes a localized reflexive safety routine (emergency stop or hard steering correction), and safely re-syncs once the path is clear.

## 3. THE 3D ENVIRONMENT PIPELINE

We avoid complex collision mesh mapping in CAD/Blender by using a hybrid asset pipeline:
- **Structural Walls & Openings:** Modeled directly inside Gazebo using the Gazebo Building Editor. A 2D house floor plan is imported, traced, and extruded directly into a `.world` file to ensure lightweight, mathematically perfect physical collision boundaries.
- **Interior Obstacles & Furniture:** Complex models (couches, tables, custom obstacles) are designed or modified in Blender, exported cleanly as `.dae` (Collada) files, and imported as static models directly into the pre-built Gazebo world layout.
